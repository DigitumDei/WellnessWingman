## Executive Summary

This branch extends the data migration (backup/restore) feature to include nutritional profiles and their associated source images, wires the migration service to the profile repository, and surfaces profile import counts in the settings UI. All three reviewers converged on one concrete correctness defect after critique: when a desktop user picks an image from outside appDataDir, the profile's sourceImagePath is stored as an absolute path. On export the JSON retains that absolute path unchanged, the ZIP stores the file under its bare basename, and on import resolveImportPath leaves the absolute path intact — so the restored profile points to a stale path on the source machine even though the file was physically copied into the archive. This regression is grounded in CameraCaptureService.desktop.kt returning selectedFile.absolutePath for gallery picks, meaning it is triggered by the app's own normal desktop usage. The verdict is revise until this path round-trip is fixed and the missing test coverage is added.

## Blocking Issues

- External sourceImagePath not rewritten on import for desktop gallery images (shared/src/commonMain/kotlin/com/wellnesswingman/domain/migration/DataMigrationService.kt): relativizePath leaves any path outside appDataDir unchanged in the JSON. addFileToExport stores the copied file under its basename only. resolveImportPath receives the original absolute path from JSON and returns it as-is because it is already absolute. The imported profile therefore points to a stale source-machine path even though the image was packaged and extracted correctly. CameraCaptureService.desktop.kt returns selectedFile.absolutePath for gallery picks, so this is triggered by normal desktop usage.
- Basename collision for external images with identical filenames (shared/src/commonMain/kotlin/com/wellnesswingman/domain/migration/DataMigrationService.kt): addFileToExport uses substringAfterLast('/') as the ZIP entry name for any path outside appDataDir. Two profiles referencing different external files that share a filename (e.g. /user/a/label.jpg and /user/b/label.jpg) will collide: one entry is silently dropped via the exportedPaths deduplication set, and the surviving path in JSON still points to a non-existent location after import.

## Non-Blocking Issues

- resolveImportPath called during export — misleading name with no explanatory comment (shared/src/commonMain/kotlin/com/wellnesswingman/domain/migration/DataMigrationService.kt): The helper is named for the import direction but is invoked during export file enumeration. The behavior is intentional but unintuitive; a comment at the call site explaining why is the minimum fix.
- Backward-compatibility test only asserts the new field is absent, not that legacy data still imports (shared/src/commonTest/kotlin/com/wellnesswingman/domain/migration/DataMigrationServiceTest.kt): importData preserves backward compatibility when nutritional profiles are absent checks nutritionalProfilesImported == 0 but does not assert that the entry present in the archive was actually imported, making it an incomplete regression guard.
- Fake profile repository does not simulate INSERT OR REPLACE uniqueness constraints (shared/src/commonTest/kotlin/com/wellnesswingman/domain/migration/DataMigrationServiceTest.kt): The test double simply appends to a list. The real SqlDelightNutritionalProfileRepository uses INSERT OR REPLACE with unique constraints on both profileId and externalId. Conflict behavior on re-import is entirely untested.
- Import test name is stale after nutritional profile coverage was added (shared/src/commonTest/kotlin/com/wellnesswingman/domain/migration/DataMigrationServiceTest.kt): importData imports entries, analyses, summaries, and weight records now also covers nutritional profiles (import count, upsert, image path resolution) but the name was not updated.

## Missing Tests

- Import round-trip where sourceImagePath is an absolute path outside appDataDir: verify the imported profile's path is rewritten to an appDataDir-relative location and the file exists there.
- Export with two profiles whose external images share the same basename from different directories: verify both files are preserved distinctly in the archive and both profiles import with correct paths.
- Import loop error accumulation: one malformed profile should not abort the remaining profiles, and result.isSuccess should be false while entriesImported reflects the successful count.
- Export and import of a profile where sourceImagePath is null or blank: verify the guard clause is exercised and no file operations are attempted.
- Export where the source image file does not exist on disk: verify the profile still appears in the JSON with its path and addFileToExport's silent-skip behavior is explicitly asserted.
- Import against a repository fake (or integration test) that simulates INSERT OR REPLACE uniqueness on profileId and externalId: verify re-import and conflict resolution behave as expected.
