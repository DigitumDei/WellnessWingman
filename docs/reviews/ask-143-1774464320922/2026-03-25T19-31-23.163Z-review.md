## Blocking Issues

- Weekly summary entry counts are internally inconsistent (shared/src/commonMain/kotlin/com/wellnesswingman/domain/analysis/WeeklySummaryService.kt): `otherCount` includes Polar step/nightly-recharge days, but `totalEntries` does not include those same Polar contributions. This can produce impossible payloads such as `otherCount > totalEntries`, and the inconsistent values are then propagated into prompts and stored summary data.

## Non-Blocking Issues

- Default `PolarInsightService()` is a silent no-op footgun (shared/src/commonMain/kotlin/com/wellnesswingman/domain/polar/PolarInsightService.kt): Production DI appears to inject the real dependency, but the default constructor quietly falls back to an empty repository. That makes direct construction in tests or future callers easy to misconfigure without noticing.
- Polar sync error logging lost throwable context (shared/src/commonMain/kotlin/com/wellnesswingman/domain/polar/PolarSyncOrchestrator.kt): The new logging path records only a sanitized message and drops the throwable, which reduces stack-trace visibility and makes sync failures harder to diagnose.
- Test fixtures are duplicated across summary tests (shared/src/commonTest/kotlin/com/wellnesswingman/domain/analysis): `FakePolarSyncRepository` is duplicated in multiple test files, which increases drift risk and makes future maintenance harder.

## Missing Tests

- Daily mixed-data precedence coverage, especially tracked exercise with supplemental Polar data on the same day
- Weekly precedence coverage for `trackedExerciseDates` / `trackedSleepDates` suppression behavior
- Weekly count-integrity coverage for Polar-only step/nightly-recharge contributions and `otherCount` vs `totalEntries` consistency
- Polar-only day-detail UX coverage proving the screen succeeds, shows Polar metrics, and exposes the summary action card

## Disputed Issues

- Whether day-level tracked-vs-Polar suppression is a correctness bug or an acceptable v1 design choice. One review argues it violates the stated 'tracked entries take priority, Polar fills gaps' intent by dropping unrelated same-day Polar sessions; the other treats day-level precedence as a reasonable product decision rather than a proven defect.

## Outstanding Concerns

- The regex-based log redaction approach may deserve broader hardening, but no concrete escaping payload was agreed on in review.

## Blocking Issues

1. **`formatHours` truncates instead of rounding, and is wrong for edge cases.** In `PolarDayContext.formatHours`, the formula `(seconds / 360.0).toInt() / 10.0` divides by 360 instead of 3600 first. Wait — let me re-check: `seconds / 360.0` gives tenths-of-hours, then `.toInt()` truncates, then `/ 10.0` gives hours to one decimal. That's actually correct mathematically (e.g., 28800 / 360 = 80, 80 / 10 = 8.0). But `formatMinutes` uses `seconds / 6.0` which gives tenths-of-minutes, same pattern. These are correct but needlessly obscure. **Not blocking after all — retracted.**

2. **`DailySummaryService` default parameter creates a broken fallback.** Both `DailySummaryService` and `WeeklySummaryService` have `polarInsightService: PolarInsightService = PolarInsightService()`. The no-arg `PolarInsightService()` constructor uses `EmptyPolarSyncRepository` which is a private object in `PolarInsightService.kt`. This means the default is a silent no-op. While DI wires the real one, any code path that constructs the service directly (tests, or future callers) silently gets no Polar data with no warning. The DI module already passes the real instance, so the default is technically dead code in production — but it's a footgun. **This should be constructor-required (no default) so callers are forced to be explicit.** If the default is intentional for backwards compatibility, it should at minimum log a warning.

3. **`totalSteps` uses `maxOfOrNull` which is semantically wrong for multi-activity days.** If there are multiple `StoredPolarActivity` records for a single day (e.g., synced twice), `maxOfOrNull { it.data.totalSteps }` picks the highest one. If the API returns partial syncs or updates, this might be correct (latest has highest steps). But if there are genuinely separate activity records (unlikely but possible), this silently drops data. `sumOf` or taking the latest-by-timestamp would be more defensible. **Weak concern** — depends on Polar API contract — but worth validating.

4. **Mutable `var totalSleepHours` mutation is fragile.** In `DailySummaryService.kt`, `totalSleepHours` is declared earlier in the function (not shown in diff) and then conditionally overwritten:
   ```kotlin
   if (includePolarSleep && totalSleepHours == null) {
       totalSleepHours = polarContext.sleepDurationHours
   }
   ```
   This depends on `totalSleepHours` being `null` when there's no tracked sleep. If the upstream calculation ever changes to default to `0.0` instead of `null`, Polar sleep silently disappears. The condition `includePolarSleep` already checks `sleepCount == 0`, so the `totalSleepHours == null` guard is redundant-but-fragile. **Minor but worth a comment or simplification.**

## Non-Blocking Issues

1. **`DailySummaryActionCard` is extracted but duplicates logic.** The `else if` branch at line 170 shows the summary action card when `entries.isEmpty()` but `summaryCardState != Hidden`. This means Polar-only days show a "Generate summary using your synced Polar data" card even if the summary card state is `NoSummary` — which is correct. However, the card text says "Generate a daily recap using your synced Polar data" unconditionally, even if the user also has tracked entries (the first branch). The first branch renders the inline `DailySummaryCard` (not `DailySummaryActionCard`), so this is fine. Clean separation.

2. **`PolarDayContext` default date is `LocalDate(1970, 1, 1)`.** In `DayDetailUiState.Loaded`, the default `polarContext` has date 1970-01-01. This is a placeholder that should never be displayed, but if it leaks into a prompt or UI, it'll look wrong. Consider making it match the actual date or making the field nullable.

3. **`nightlyRecharge.maxByOrNull { it.data.recoveryIndicator }` in the UI** — if `recoveryIndicator` can be negative or zero, `maxByOrNull` might pick unexpected records. Check that this selector is appropriate.

4. **`WeeklySummaryService.getDayContexts` date range is `weekEnd.plus(1, DateTimeUnit.DAY)`.** This means `weekEnd` is inclusive. Verify that `weekEnd` consistently means the last day of the week (not the first day of next week) across all callers. If `weekEnd` is already exclusive somewhere, you'd fetch an extra day.

5. **Duplicated `FakePolarSyncRepository`** in both `DailySummaryServiceTest` and `WeeklySummaryServiceTest`. Extract to a shared test fixture to avoid drift.

6. **`sanitizeForLogs` truncates to 160 characters.** The truncation happens *after* redaction, which is correct. But 160 chars may be too short for debugging server errors. Consider making this configurable or at least documenting the limit.

7. **Redaction regex for `refresh_token` uses `refresh[_ -]?token`.** This won't match `refresh-token` with the hyphen because `[_ -]` includes space and underscore but the hyphen is at the end of the character class — actually, `[_ -]` means "underscore, space through hyphen" which is a range. This is a regex bug: `[_ -]` means characters from space (0x20) to hyphen (0x2D), which includes `!`, `"`, `#`, `$`, `%`, `&`, `'`, `(`, `)`, `*`, `+`, `,`, `-`. The hyphen should be escaped or placed first: `[_ \\-]` or `[-_ ]`. **This works by accident** (the range does include hyphen) but is misleading and fragile.

## Missing Tests

1. **No test for tracked-exercise-takes-precedence-over-Polar-exercise.** There's a test for sleep precedence but not exercise. The logic is symmetric, but the test should exist.

2. **No test for `PolarDayContext.buildPromptLines`.** The prompt line formatting (steps, sleep, exercise, nightly recharge) is untested directly. The integration tests check that certain strings appear in prompts, but edge cases (zero calories, zero distance, blank training benefit) aren't covered.

3. **No test for `PolarInsightService.getDayContexts` grouping logic.** The weekly batch query groups by date — verify correct behavior when data spans date boundaries.

5. **No test for `WeeklySummaryService` precedence** — the weekly service has date-level tracked-vs-Polar precedence logic (`trackedExerciseDates`, `trackedSleepDates`) that is untested.

6. **`sanitizeForLogs` edge case: null input returns empty string** — tested implicitly but not explicitly.

## Strong Concerns

1. **The default `PolarInsightService()` constructor with `EmptyPolarSyncRepository` is the most architecturally concerning part.** It means `DailySummaryService` and `WeeklySummaryService` can be constructed without DI and silently produce incomplete results. In a Koin-wired app this is unlikely to bite, but it violates fail-fast principles. If someone writes a test and forgets to pass the polar service, they get green tests that don't test Polar at all — with no indication.

2. **The `otherCount` calculation in `WeeklySummaryService` now adds Polar records:**
   ```kotlin

   ```
   This mixes Polar step/recharge data into the "other" count, which is conceptually wrong. Steps and nightly recharge aren't "other entries" — they're supplemental device metrics. This inflates `otherCount` and `totalEntries` in the stored `WeeklySummary`, making historical comparisons inaccurate once Polar is connected.

## Confidence

**Medium-high.** The core architecture (bridge service, precedence rules, prompt attribution) is sound and well-thought-out. The DI wiring, UI changes, and test coverage are reasonable for a first pass. The main risks are: (1) the default no-op constructor hiding misconfiguration, (2) the `otherCount` inflation, and (3) the regex character class that works by accident. None of these are likely to cause production failures immediately, but items 1 and 2 should be addressed before merge.
