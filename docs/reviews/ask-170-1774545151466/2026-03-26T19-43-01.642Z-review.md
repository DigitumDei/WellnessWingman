## Executive Summary

The PR adds LLM tool-calling support with a registry-driven, bounded multi-turn loop for OpenAI and Gemini providers. The architectural approach is sound, but the implementation has three blocking defects: two layers of CancellationException swallowing (in OpenAiLlmClient and AnalysisOrchestrator) that prevent coroutine cancellation from propagating end-to-end, and a Gemini deserialization gap where malformed tool arguments crash the analysis with no recovery. These directly contradict the stated production-hardening goals of the change.

## Blocking Issues

- OpenAiLlmClient runCatching swallows CancellationException (shared/src/commonMain/kotlin/com/wellnesswingman/domain/llm/OpenAiLlmClient.kt): The runCatching block around tool argument parsing and executor invocation catches all Throwable including CancellationException, silently converting cancellation into an error ToolResult. The conversation loop continues instead of propagating cancellation. The ToolRegistry correctly rethrows CancellationException, but this catch-all undoes it. Fix: add 'if (error is CancellationException) throw error' as the first line of the getOrElse block.
- AnalysisOrchestrator.processEntry catch-all also swallows CancellationException (shared/src/commonMain/kotlin/com/wellnesswingman/domain/analysis/AnalysisOrchestrator.kt): The catch (e: Exception) block catches CancellationException (which extends Exception in Kotlin), marks the entry as FAILED, and returns an error result. Even if the OpenAI client is fixed, cancellation still cannot propagate past the orchestrator. Fix: add 'if (e is CancellationException) throw e' before the error handling, or use coroutineContext.ensureActive().
- Gemini client has no recoverable path for malformed tool arguments (shared/src/commonMain/kotlin/com/wellnesswingman/domain/llm/GeminiLlmClient.kt): GeminiFunctionCall.args is hard-typed as JsonObject. If the Gemini API returns non-object args (null, string, array), httpResponse.body<GeminiResponse>() throws a deserialization exception before any tool-level error handling can intervene. The OpenAI client handles the analogous case via runCatching around JSON parsing, making the Gemini client strictly less robust. This contradicts the stated goal of handling malformed tool arguments without crashing.

## Non-Blocking Issues

- N+1 query pattern in get_recent_entries tool (shared/src/commonMain/kotlin/com/wellnesswingman/domain/llm/ToolRegistry.kt): Each returned entry triggers a separate getLatestAnalysisForEntry database call inside a map loop. Bounded at 10 entries so not a performance emergency, but a JOIN or batch fetch would reduce latency inside the multi-turn tool loop.
- EntryType.fromString silently maps invalid LLM input to UNKNOWN (shared/src/commonMain/kotlin/com/wellnesswingman/domain/llm/ToolRegistry.kt): If the LLM passes an unrecognized entryType string, fromString returns UNKNOWN without signaling an error. The query then filters by entryType = UNKNOWN, returning unrelated or empty results while the tool response looks successful. The LLM has no signal that its filter was ignored.

Gemini claims both clients over-count prompt tokens by summing `promptTokens`/`promptTokenCount` across turns, arguing that LLM APIs report *cumulative* prompt tokens per request. This claim is **not well-supported**:

- **OpenAI**: The `openai-kotlin` SDK's `Usage.promptTokens` reports tokens for *that specific request's* prompt. When conversation history grows, the next request's `promptTokens` naturally includes prior messages, but each reported value is the token count for that request, not a running total of all requests. Whether summing is "correct" depends on what the project wants to measure: total prompt tokens billed (sum is correct) vs. unique prompt tokens processed (final value is closer). The claim that "the final `promptTokens` should be the `promptTokenCount` from the *final* turn" would *undercount* billing if used that way.
- **Gemini**: `usageMetadata.promptTokenCount` similarly reports tokens for that request. The same ambiguity applies.
- Gemini's review provides no evidence for how these specific APIs report token counts, and the "correct" approach depends on whether the project is tracking billing or unique processing. Calling this a "blocking" bug with "high confidence" is overstated without establishing what the token counts are intended to measure and how these specific APIs behave.

**Verdict: Weak claim presented with false confidence. Needs evidence before it can be called blocking.**

**2. Gemini's "Gemini Response Sanitization" concern — WEAK**

The model's content (including text) is preserved in the conversation history at line 208 (`contents.add(content.copy(role = "model"))`), so it's not lost. The concern about "missing reasoning steps in logs" is speculative and low-impact. **Overstated.**

**3. Codex's "No repository-level test for SQLDelight getRecentEntries" — MARGINAL**
This is a reasonable observation but weak as a review finding. SQLDelight generates type-safe queries from SQL; the generated code is deterministic. Testing the generated query against a real database is integration testing, not a gap specific to *this* PR. **Not wrong, but out of scope for this review.**

## Missing Context

If the executor throws any exception (not just `CancellationException`), the entire `runConversation` call fails with no tool-error recovery. The `ToolRegistry.execute()` does catch non-cancellation exceptions (lines 68-79), but if the issue is *before* the registry (e.g., in argument extraction from `functionCall.args`), there's no protection. This is arguably a more fundamental gap than the OpenAI `runCatching` issue — the OpenAI client at least *tries* to recover (and accidentally catches too much), while the Gemini client has no recovery at all at this layer.

**2. No reviewer noted the Gemini client's `functionCall.id` may be null**
`GeminiFunctionCall.id` is typed as `String?` (line 313), and it's passed directly to `ToolCall(id = functionCall.id, ...)` at line 215.
