## Executive Summary

The branch adds a nutritional profile workflow (scan, save, manage, LLM tool integration) with a sound architectural foundation, but has three blocking defects: the profiles list doesn't refresh after save/edit (breaking the core management workflow), the Gemini provider's system instruction actively steers the model away from the new tools, and the get_nutritional_profiles tool schema omits the required field for profileIds. Several non-blocking UX and robustness issues also need attention, and test coverage for the new components is thin.

## Blocking Issues

- Profiles list does not refresh after save/edit (composeApp/src/commonMain/kotlin/com/wellnesswingman/ui/screens/nutrition/NutritionalProfilesViewModel.kt): NutritionalProfilesViewModel loads profiles once in init. After saving or editing a profile in NutritionLabelScanScreen, navigator.pop() returns to the already-instantiated Voyager ScreenModel with stale data. Users will think their save failed. Nothing triggers loadProfiles() on return.
- Stale Gemini system instruction contradicts new tool flow (shared/src/commonMain/kotlin/com/wellnesswingman/domain/llm/GeminiLlmClient.kt): GeminiLlmClient.kt line 331 hardcodes the system instruction to say tools are limited to 'profile, weight history, and recent tracked entries.' The new list_nutritional_profiles and get_nutritional_profiles tools are registered but the instruction actively discourages the model from calling them, effectively breaking the nutritional profile feature for Gemini users.
- get_nutritional_profiles schema missing required field for profileIds (shared/src/commonMain/kotlin/com/wellnesswingman/domain/llm/ToolRegistry.kt): ToolRegistry.kt defines profileIds as a property but never declares "required": ["profileIds"] in the JSON schema. The runtime handler rejects calls without it, but tool-calling models rely on the schema contract to determine which parameters to include. This weakens the two-step list-then-get flow the feature depends on.

## Non-Blocking Issues

If a second scan fails, old extracted values from a different image remain in the form and can be saved.

Minor but easy to fix and avoids any ambiguity in LLM interpretation.

### Claude (claude / claude-opus-4-6)

```text
## Blocking Issues

**1. Duplicate guideline numbering in analysis prompt**

In `AnalysisOrchestrator.kt`, after inserting guideline #4 for saved packaged foods, the original guideline #4 (for exercise) is not renumbered. The prompt now has two items numbered "4". This could confuse the LLM and cause it to skip the exercise guideline.

```
+4. For meals with matching saved packaged foods, use the lookup tool and mark those nutrition values as `exact`
 4. For exercise: Extract metrics from fitness tracker screenshots or estimate from images
```

**2. `get_nutritional_profiles` schema missing `required` field**

The `parametersSchema` for `get_nutritional_profiles` in `ToolRegistry.kt` defines `profileIds` as a property but never declares it as required. LLMs may omit the field entirely, hitting the error branch silently. The schema should include `"required": ["profileIds"]`.

**3. `NutritionLabelScanUiState` uses `ByteArray` in a `data class` — broken equality/copy semantics**

`NutritionLabelScanUiState` is a `data class` with a `ByteArray?` field (`photoBytes`). `ByteArray.equals` uses reference equality, so `StateFlow` emissions that only differ in photo bytes content (but are different array instances) won't trigger recomposition, and `copy()` comparisons will be wrong. This is a latent bug that will bite during image re-capture or reload.

## Non-Blocking Issues

**1. `intOrNull` narrowing in `get_nutritional_profiles` handler**

The code uses `it.jsonPrimitive.intOrNull?.toLong()` to parse profile IDs. If the LLM sends a `Long` value that overflows `Int`, it will silently drop it. Should use `longOrNull` instead.

**2. `list_nutritional_profiles` fetches all profiles unconditionally**

`getAll()` loads every profile into memory. Fine for a handful of profiles, but there's no limit. If a user accumulates hundreds of profiles, this becomes a performance issue for every meal analysis LLM call. Consider adding a limit or paginating.

**3. `FakeNutritionalProfileRepository` duplicated in three test files**

The same fake implementation is copy-pasted in `AnalysisOrchestratorTest`, `DailySummaryServiceTest`, and `WeeklySummaryServiceTest`. Should be extracted to a shared test fixture.

**4. `upsert` uses `INSERT OR REPLACE` which resets `AUTOINCREMENT` rowid behavior**

When `profileId = 0` is passed to `upsert`, `INSERT OR REPLACE` with `profileId = 0` will insert a row with `profileId = 0` rather than auto-generating an ID, because 0 is a valid explicit value. The `insert` path avoids this by not including `profileId`. The `upsert` method could produce unexpected results if called with a default-constructed profile.

**5. No navigation refresh after save in `NutritionLabelScanScreen`**

After saving a profile, the scan screen calls `navigator.pop()`. The `NutritionalProfilesScreen` loads profiles only in `init`. If the profiles screen is already on the back stack, it won't refresh to show the newly saved/edited profile. The list will appear stale until the user navigates away and back.

**6. Source display logic is fragile**

In `EntryDetailScreen.kt`, the source display checks `source.equals("exact", ignoreCase = true)` with a hardcoded string. If the LLM returns any other value (e.g., "exact_profile", "saved"), it falls through to "Estimated by AI" — which is misleading. A missing source (`null`) is handled separately (the `let` skips it), but a non-null, non-"exact" value like "vision" would show "Estimated" which happens to be correct, yet this is coincidental rather than robust.

2. **No test for `SqlDelightNutritionalProfileRepository`** — the `searchByName` query with its ranking logic, `toDomain` mapping, alias encoding/decoding, and `upsert` behavior are untested.
3. **No test for `NutritionLabelScanViewModel`** — save validation (blank name rejection), alias parsing, edit-mode profile loading, and capture/analyze flows are untested.
4. **No test for `NutritionalProfilesViewModel`** — load and delete flows are untested.
5. **No integration test verifying the analysis prompt actually triggers tool calls** — the prompt instructs the LLM to call `list_nutritional_profiles` then `get_nutritional_profiles`, but there's no test verifying the orchestrator handles a multi-turn tool-use conversation correctly with these new tools.

## Strong Concerns

The `searchByName` SQL query in `NutritionalProfile.sq` uses named parameters (`:query`, `:aliases`, `:exact`, `:prefix`) but the repository method in `SqlDelightNutritionalProfileRepository.searchByName` passes `contains` for both `:query` and `:aliases`. This works for the `LIKE` clauses but the parameter naming is slightly confusing — `aliases` parameter receives the same `%query%` pattern as `query`. More importantly, the alias search does `LIKE` against the raw JSON string (`'["protein bar","quest bar"]'`), which means a search for `bar` would match inside JSON syntax (brackets, quotes). This is functional but brittle — a search for `]` or `"` would match every row.

The prompt change is the riskiest part: it asks the LLM to make tool calls *during* image analysis. I don't see evidence in this diff that the orchestrator's `analyzeMealEntry` / image analysis flow actually handles tool call responses in a loop. If it doesn't, the LLM will request a tool call and the orchestrator will treat it as the final response, failing to parse it as JSON analysis. **This would be a silent regression for all meal analyses when profiles exist**, since the prompt now unconditionally instructs the LLM to call tools first.

## Confidence

**Medium-low.** The data layer, UI, and test plumbing look solid. But the critical integration point — whether the orchestrator actually handles the multi-turn tool-use flow the prompt now demands — is not verifiable from this diff alone, and if it doesn't, this change breaks meal analysis rather than enhancing it. The duplicate guideline numbering and missing `required` in the schema are straightforward fixes. The `ByteArray` equality issue is a real but low-severity bug.
```
