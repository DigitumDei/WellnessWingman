## Issues Found

### 1. Prompt references tool that might not be registered (Medium)

**Location:** `AnalysisOrchestrator.kt` prompt (line ~198), `ToolRegistry.kt` (line ~193)

The unified analysis prompt always tells the LLM:

> "If the content is a meal and one or more food items appear to be packaged foods with saved exact nutrition data, call the `lookup_nutritional_profile` tool..."

But `ToolRegistry.registerNutritionalProfileTools()` only registers the tool when `nutritionalProfileRepository != null`. If the repository is absent, the LLM calls a tool that returns "not registered" — a confusing error the user would never understand.

**In practice** `DomainModule` always injects the repository, so this can't happen today. But the coupling is fragile — the prompt and registration should either both be conditional or both be unconditional.

**Recommendation:** Since `DataModule` always provides `NutritionalProfileRepository`, make the constructor parameter non-optional in `ToolRegistry` to enforce this guarantee at compile time.

---

### 2. No confidence threshold on label extraction (Medium)

**Location:** `NutritionLabelAnalyzer.analyzeLabelImage()`, `NutritionLabelScanViewModel.analyzeImage()`

The analyzer returns `NutritionLabelExtraction` with a `confidence` field, and `extractionWarnings` are surfaced to the user — but nothing prevents saving a 0.1-confidence extraction. The user sees warnings but the Save button is always enabled.

**Recommendation:** Either disable Save below a threshold (e.g., confidence < 0.3), or add a confirmation dialog: "Low confidence extraction — are you sure you want to save?"

---

### 3. String-based "exact" source matching is fragile (Medium)

**Location:** `EntryDetailScreen.kt` (lines ~522, ~557), `ToolRegistry.kt` (line ~249), `MealAnalysisResult.kt`

The `source` field flows through: ToolRegistry (hardcodes `"exact"`) -> LLM prompt (instructs LLM to output `"exact"` or `"estimated"`) -> `MealAnalysisResult` deserialization (`source: String?`) -> `EntryDetailScreen` (`source.equals("exact", ignoreCase = true)`).

The LLM is a non-deterministic text generator. It might return `"Exact"`, `"exact match"`, `"profile"`, `"saved"`, or any variation. The UI comparison would silently fail and show no attribution.

**Recommendation:** Normalise at the deserialization boundary. When parsing `MealAnalysisResult` from LLM JSON, map the raw string to an enum (`EXACT` / `ESTIMATED`) using lenient matching (e.g., contains "exact" -> `EXACT`, everything else -> `ESTIMATED`). The UI then matches on the enum safely.

---

### 4. Stale extraction state on re-analysis failure (Low)

**Location:** `NutritionLabelScanViewModel.analyzeImage()` (lines 111-127)

If the user captures a second photo and analysis fails (network error, LLM timeout), the old nutrition values and warnings from the first successful analysis remain visible. The user might not notice the error and save stale data.

**Recommendation:** Clear `nutrition`, `extractionWarnings`, and `rawJson` at the start of `analyzeImage()` before setting `isAnalyzing = true`.

---

### 5. No duplicate profile detection (Low)

**Location:** `NutritionLabelScanViewModel.save()` (lines 155-205)

Nothing prevents creating multiple profiles with identical or very similar names. User scans "Quest Bar" twice by mistake, gets two profiles. The LLM lookup might then return both as matches.

**Recommendation:** Before insert, check `repository.searchByName(primaryName, limit = 1)`. If a close match exists, show a confirmation dialog or offer to edit the existing profile instead.

---

### 6. Prompt guideline numbering (Trivial)

**Location:** `AnalysisOrchestrator.kt` unified prompt

Guideline number 4 appears twice in the prompt. Should be numbered 4-7 sequentially.

## Naming Observation

The domain model uses `calories` while `NutritionalProfileNutrition` (the lookup result DTO) uses `totalCalories` (with `@SerialName`). This is intentional — `totalCalories` matches the naming convention used in `NutritionEstimate` throughout the analysis pipeline. Not a bug, but worth documenting.

Similarly, `FoodItem.matchedProfileName` is singular (one food -> one profile) while `NutritionEstimate.matchedProfiles` is plural (one meal -> multiple profiles). Semantically correct.

## Design Note: Tool-Based Lookup Strategy

The current `lookup_nutritional_profile` tool relies on SQL `LIKE` matching against the name the LLM guesses from the photo. This works but is brittle — the LLM might say "protein bar" while the saved profile is called "Quest Protein Bar Chocolate Chip".

**Embeddings** would solve this with semantic matching, but add significant overhead: an embedding model, a vector store or similarity search layer, and latency per query. For a personal food library that will realistically contain 50-200 items, this is overkill.

**Preferred alternative: two-tool pattern.** Instead of one tool that searches by name, split into:

1. **`list_nutritional_profiles`** — returns all saved profile names/aliases (lightweight, no nutrition data). With ~50-200 items this is a small payload the LLM can scan in-context.
2. **`get_nutritional_profiles`** — accepts a list of `profileId`s and returns full nutrition details for those specific profiles.

This lets the LLM do the fuzzy matching itself (it's good at recognising that "protein bar" probably means "Quest Protein Bar Chocolate Chip") without relying on SQL pattern matching. The first call is cheap (names only), the second is targeted (exact IDs). No embeddings, no vector store, and the LLM's own language understanding handles the matching.

At scale (thousands of profiles) this wouldn't work — the name list would blow the context window. But for a personal food library it's the right trade-off: simple, zero infrastructure, and leverages what the LLM is already good at.
